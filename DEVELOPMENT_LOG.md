## 2026-07-01 小程序原型设计第一轮

### 本次目标

使用 `saixiaofeng-basketball` 技能启动赛小蜂篮球小程序原型设计，重点让第一版不只是工程骨架，而是具备篮球赛事运营和计分台的真实产品形态。

### 已完成

- 创建 Taro + React + TypeScript 小程序前端工程。
- 配置首页、计分台、赛事、球队、技术统计五类页面。
- 配置 GitHub 与 Gitee 双远端。
- 生成并纳入 `package-lock.json`，固定依赖解析结果。
- 首页升级为运营工作台：今日比赛、待确认阵容、待审核赛果、快捷入口、运营提醒、今日赛程。
- 计分台升级为比赛现场工作台：比分、节次、计时、同步状态、模式切换、场上阵容、主客队加分、犯规、暂停、换人、技术统计快捷键、事件流、撤销上一条、底部比赛控制。
- 数据层补充运营指标、提醒事项、场上阵容、计分模式和技术统计动作。

### 设计原则

- 第一屏必须是可用工作台，不做营销落地页。
- 计分场景优先满足高压、快速、少误触的操作。
- 比分、时间、节次、犯规、暂停和事件流必须可快速扫描。
- 所有后续接口应围绕比赛、球队、球员、事件、赛程、赛事、统计这些结构化对象扩展。

### 下一步建议

- 接入真实接口前，先补齐比赛详情页和赛前阵容确认流程。
- 给计分台增加本地事件队列、离线重试和冲突处理状态。
- 增加赛事报名、支付、家长端观赛和分享链路。
- 用微信开发者工具跑 `npm run dev:weapp` 后做移动端真机尺寸检查。

## 2026-07-01 MVP 流程简化

### 用户反馈

首页和计分流程过于复杂，当前阶段只需要快速开赛、横屏计分、结束生成赛果，以及简单创建赛事和球员名单。

### 已调整

- 首页改为简单入口：进入比赛积分系统、最近赛果、创建赛事、球员名单。
- 比赛页改为两步流程：先输入两支队伍名称，再进入横屏计分盘。
- 计分盘支持 +1 / +2 / +3、撤销、重开、结束比赛。
- 结束比赛后写入 `latestMatchResult` 本地缓存，并自动回到首页展示最近赛果。
- 赛事页简化为赛事名称、地点、日期创建。
- 球员页简化为球员姓名和号码录入。

## 2026-07-01 横屏计分页路由修复

### 问题

用户反馈：首页进入比赛无反应，底部点击比赛后虽然横屏但页面为空白。

### 原因判断

`pages/scorer/index` 同时作为 TabBar 页面并配置 `pageOrientation: 'landscape'`，在微信开发者工具中容易出现横屏容器切换后页面主体不渲染的问题。

### 已调整

- 从底部 TabBar 移除比赛页，只保留首页、赛事、球员。
- 首页“进入比赛积分系统”改为 `Taro.navigateTo({ url: '/pages/scorer/index' })`，打开独立横屏页面。
- 首页入口从 Button 改成 View 样式按钮，减少 Button 默认行为干扰。
- 保留 `pages/scorer/index` 的 `pageOrientation: 'landscape'`，让独立计分页继续横屏。
- 已确认 `dist/app.json` 不再包含比赛 Tab，首页产物包含 `navigateTo`。
